LoRA 微调策略 peft

  • 冻结 LLM 全部 ~1.8B 参数
  • LoRA rank=16, alpha=32 → ~20M 可训练
  • 目标模块: q/k/v/o_proj + gate/up/down_proj
  • 梯度累积 4 步，等效 batch=256
  • AdamW lr=2e-4, OneCycleLR 调度
  • 保存: LoRA 权重 + 任务头权重分开存储

特征序列化 vs 拼接 设计变更

旧方案: 各特征 Embedding → Concat → MLP → 256d

新方案: 各特征 → 投影为 LLM token → 自注意力交互

  • [USER] user_feat → 1 token (2048d)
  • [AD] ad_feat → 1 token
  • [CTX] ctx_feat → 1 token
  • [BEH_S] beh_1..beh_50 [BEH_E] → 50 tokens
  • [CLS] → 聚合所有语义的最终表示
  • LLM 自注意力自然建模特征交叉

RetrievalHead 简化 设计变更

  • 旧: 独立 TransformerDecoder 2层8头
  • 新: 轻量 SID Embedding + Linear 输出
  • 复用 LLM 的深层语义能力
  • user_repr (2048d) 作为条件注入
  • 训练: BOS + teacher forcing
  • 推理: get_next_token_logits() → STATIC 掩码

Fallback 模式 兼容

当无法下载混元模型时自动降级:

  • 设置 model_name_or_path: "fallback"
  • 创建内置 6层 TransformerEncoder
  • hidden_size=2048，接口完全一致
  • 适合本地开发/CI测试/CPU 演示
  • 生产部署时切换为真实模型路径

generate_and_apply_logprobs_mask() STATIC核心

论文的核心内核，与 backbone 无关：

  • starts = csr_indptr[flat_states] — 查行起始
  • actual_lens = indptr[state+1] - starts
  • offsets = arange(limit) — 偏移网格
  • gathered = packed_csr[starts + offsets] — 突发读取
  • valid_mask = offsets < actual_lens
  • logprobs.gather(1, token_ids)
  • 无效路径置为 -inf
  • 复杂度: O(1) 相对于约束集大小 |C|

接口兼容性 设计保证

两种 backbone 共享完全相同的推理接口：

  • model.get_user_representation(batch) → (B, H)
  • model.retrieval_head.get_next_token_logits(user_repr, partial_sids) → (B, 1, V)
  • model.forward(batch) → {ctr_prob, cvr_prob, ...}
  • STATICConstrainedDecoder 无需修改
  • DSPBidEngine 无需修改
  • 切换 backbone 只需改 config

合并前向推理 必选

  • 原始: get_user_representation() + forward() = 2次 LLM 前向
  • 优化: merged_inference() = 1次 LLM 前向
  • 一次出 CTR + CVR + user_repr
  • DSPBidEngine 自动适配
  • 延迟降低约 50%，零精度损失

INT8 量化策略 核心

  • 动态量化: 无需校准数据，开箱即用
  • 静态量化: 需校准数据，精度更高
  • bitsandbytes: LLM.int8() GPU 原生
  • GPTQ: 离线量化，需预处理
  • 配置: optimization.quantization_method

TensorRT INT8 最佳性能

  • 路径: PyTorch → ONNX → TensorRT Engine
  • INT8 + FP16 回退，自动校准
  • 动态 batch: 1 ~ 32
  • 工作空间: 2GB
  • 目标: p99 ≤ 8-12ms (A10 GPU)

推荐部署方案 场景

  • 生产环境: 合并前向 + INT8 + TensorRT → p99 ≤10ms
  • 开发测试: 合并前向 + torch.compile → p99 ≤20ms
  • CPU 部署: 合并前向 + ONNX Runtime → p99 ≤50ms

层裁剪 收益最大

混元 1.8B 共 32 层 Transformer，推理时裁剪到 6 层可减少 75% 计算量。

  • 配置: optimization.num_inference_layers: 6
  • 原理: CTR/CVR 任务对深层语义不敏感
  • AUC 下降 < 1% (广告系统可接受)
  • 单独使用即可获得 3-4x 加速
  • 与 LoRA 微调兼容 (只影响推理)

动态批处理 吞吐关键

GPU 的并行计算能力在 batch=1 时严重浪费，批处理可大幅提升吞吐。

  • batch=1: QPS ≈ 80 (GPU 利用率 < 10%)
  • batch=8: QPS ≈ 500 (8x 请求, 延迟 +30%)
  • batch=16: QPS ≈ 800 (GPU 利用率 ~60%)
  • batch=32: QPS ≈ 1200 (接近峰值吞吐)
  • 生产方案: 微批聚合 + 最大等待时间 5ms

多卡并行推理 扩展方案

单卡优化到极致后，通过多卡并行线性扩展 QPS。

  • DataParallel: 多卡各自处理不同 batch
  • N 卡 = N × 单卡 QPS
  • Pipeline Parallel: 模型切分到多卡
  • 降低单请求延迟 + 提升吞吐
  • 2x A10: QPS ≈ 6000-16000
  • 配合负载均衡器分发请求

python run_pipeline.py --stage ab_test

  • 自动对比: 混元 LLM vs DeepFM (效果参照)
  • 测量: 原始延迟 → 优化后延迟 → 批处理吞吐
  • 评估: AUC/LogLoss/校准度/竞价效果
  • 输出: QPS 优化路径和组合效果预估
  • 架构: 召回+排序均使用混元 LLM

V100 硬件特性 SM 7.0

  • FP16 Tensor Core: 120 TFLOPS
  • FP32: 15.7 TFLOPS (7.6x 差距)
  • 显存带宽: 900 GB/s (HBM2)
  • ⚠ 不支持 Flash Attention 2 (需 SM80+)
  • ⚠ INT8 Tensor Core 效率低 → 用 FP16
  • 配置: attn_implementation: "sdpa"

推荐配置 V100 最优

  • 精度: FP16 (V100 Tensor Core 加速)
  • 层裁剪: 32→6层 (收益最大)
  • 前向: MergedForwardWrapper
  • 批处理: batch=8~16 (性价比最优)
  • seq_len: behavior 20 (已优化)
  • 不建议: INT8 / torch.compile (V100 兼容差)

工业级场景评估 目标

  • 搜索广告: 需 QPS ≥ 1000 → L3+bs=4
  • 品牌广告: 需 QPS ≥ 500 → L3+bs=1
  • 延迟目标: P99 ≤ 20ms → L2+ 即可达标
  • 多卡扩展: 2x V100 → QPS 翻倍
